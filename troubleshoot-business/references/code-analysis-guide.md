# 代码逻辑分析向导

本文件是代码逻辑分析维度的指南。独立维护，可单独更新。

## 分析目标

通过阅读代码，理解业务流程，找到关键入口日志和可能的分支路径，为后续日志查询提供搜索依据。

---

## 分析流程

### 第一步：确认代码位置

1. **【必须执行】询问用户代码目录** — 必须调用 `AskUserQuestion` 工具，不可跳过、不可默认使用当前目录：
   ```
   AskUserQuestion({
     questions: [{
       question: '请提供需要分析的代码目录路径（绝对路径或相对路径均可）：',
       header: '代码目录',
       options: [
         { label: '当前工作目录', description: '<当前工作目录路径>' },
         { label: '自定义路径', description: '手动输入代码目录路径' }
       ],
       multiSelect: false
     }]
   })
   ```

2. 用户选了"自定义路径" → 要求用户输入具体路径
3. 用户选了"当前工作目录" → 使用当前工作目录
4. **检查代码图谱工具可用性**：
   - 检测当前环境是否有 `codegraph` 相关工具（如 CodeGraph MCP server）
   - 如果**没有** codegraph，提示用户：
     > 💡 当前环境未检测到 CodeGraph 代码图谱工具。CodeGraph 基于 tree-sitter 解析代码结构，可以毫秒级定位符号定义、调用关系和影响范围，比 grep 搜索更快更准确。
     > 如果需要安装，可以运行：`npm install -g @anthropic/codegraph`，然后在项目目录执行 `codegraph init -i` 初始化索引。
     > 当前将使用 grep/glob 等基础工具继续分析，功能不受影响，只是搜索效率会低一些。
   - 如果**有** codegraph → 优先使用 codegraph 进行结构性搜索（符号查找、调用链追踪等）
5. 使用当前可用的工具（codegraph、grep、glob 等）辅助分析

### 第二步：定位业务入口

根据用户描述的问题，定位相关的代码入口：

1. **理解问题描述**：用户会描述一个业务现象，如"用户 XXX 分配成功后为啥没有创建 sessionId"
2. **关键词提取**：从问题中提取关键词，如"分配"、"sessionId"、"创建"
3. **搜索入口点**：在代码中搜索相关的类、方法、注解
   - MQ 消费者：搜索 `@RabbitListener`、`@KafkaListener`、`Consumer` 等
   - API 接口：搜索 `@RequestMapping`、`@PostMapping` 等
   - 定时任务：搜索 `@Scheduled`、`CronTask` 等
   - 事件监听：搜索 `@EventListener`、`ApplicationListener` 等

### 第三步：分析入口日志

找到入口后，重点关注：

1. **入口日志**：入口方法开头通常有日志记录，如：
   ```java
   log.info("SessionAssignStatusChangeConsumer received msg: {}", msgText);
   ```
   记录下这条日志的完整文本，后续用于日志查询。

2. **参数日志**：方法入参的打印日志

3. **关键节点日志**：方法中各个判断分支处的日志

### 第四步：梳理分支路径

分析入口方法的完整逻辑，梳理所有可能的分支：

1. **正常路径**：业务正常执行的完整流程
2. **跳过路径**：各种条件判断导致提前返回/跳过的情况，如：
   - 消息数据为空
   - 消息中缺少必要字段
   - 非当前业务线的事件
   - 重复处理（会话已处理）
   - 条件不满足（如未找到匹配的客服组）
3. **异常路径**：try-catch 中的异常处理

每个分支都记录：
- 进入条件（什么情况下走这条分支）
- 相关日志（该分支输出的日志文本）
- 最终结果（对业务的影响）

### 第四步（补充）：检测线程/异步执行模式

在梳理分支的同时，**必须检测**代码中是否存在线程池或异步执行模式。这直接影响后续日志查询策略。

**检测以下模式**：

| 模式 | 关键词/注解 |
|------|------------|
| 异步注解 | `@Async` |
| CompletableFuture | `CompletableFuture.supplyAsync()`、`runAsync()`、`thenApply()`、`thenAccept()` |
| 线程池提交 | `ExecutorService.submit()`、`execute()`、`invokeAll()` |
| Spring 线程池 | `ThreadPoolTaskExecutor`、`@EnableAsync` |
| 原生线程 | `new Thread()`、`Runnable`、`newFixedThreadPool()`、`newCachedThreadPool()` |
| 并行流 | `parallelStream()`、`parallel()` |
| 定时异步 | `@Scheduled` + 线程池 |

**检测结果的影响**：

- **无论是否检测到异步模式** → 日志查询都**优先用追踪字段（TID/SID）**查询
- **检测到异步模式** → 在分析结论中标注 `⚠️ 存在异步执行，TID 可能无法串联完整链路`，提示日志查询阶段：
  - 仍先用 TID 查询，拉出主线程链路
  - 如果链路不完整（缺失异步线程日志），降级用 SID 查询
  - 最后才用业务ID + 关键字模糊查询

### 第五步：整理分析结论

输出一份结构化的分析结果：

```
【入口信息】
- 入口类/方法：<完整类名>#<方法名>
- 入口日志：<日志文本模板>
- 搜索关键字：<用于日志查询的关键字>

【异步/线程检测】
- 是否存在异步执行：是/否
- 异步模式：<具体模式，如 @Async、CompletableFuture、ExecutorService 等>
- 查询策略：策略一（TID 优先）/ 策略二（SID/业务ID 优先）
- ⚠️ 如果存在异步：标注 TID 无法串联完整链路，需用 SID 或业务ID

【可能的分支】
1. 正常流程：<描述>
   - 日志：<正常流程中的日志>
2. 跳过原因1：<原因描述>
   - 条件：<触发条件>
   - 日志：<该分支的日志>
3. 跳过原因2：<原因描述>
   - 条件：<触发条件>
   - 日志：<该分支的日志>
...

【需要日志查询确认】
- 通过代码无法确定实际走了哪条分支
- 需要查询日志系统确认
- 建议搜索关键字：<入口日志文本> + <业务ID>
```

## 分析技巧

### 从入口日志出发

入口日志是最关键的线索，因为它记录了：
- 消息/请求的原始内容
- 可以从中判断消息是否完整、字段是否齐全
- 可以提取 TID 用于后续链路追踪

### 关注条件判断

重点关注代码中的 `if` 判断，每个 `if` 都是一个可能的分支：
```java
if (StringUtils.isEmpty(msg.getId())) {
    log.warn("消息中无id字段，跳过");
    return;
}
```
这类日志是定位问题的直接证据。

### 关注 catch 块

异常处理中的日志往往包含错误原因：
```java
catch (Exception e) {
    log.error("处理分配事件异常", e);
}
```

### 关注业务状态字段

业务对象中的状态字段（如 `sessionStatus`、`waitAssignReason`）直接反映业务结果。

---

## 与日志查询的衔接

代码分析完成后，需要将结论传递给日志查询阶段：

1. **入口日志文本** → 作为 `match_phrase` 的搜索条件
2. **业务ID** → 作为 `match_phrase` 的搜索条件
3. **分支日志文本** → 用于分析日志查询结果时对照
4. **TID** → 如果从日志中提取到 TID，可进行链路追踪查询
5. **异步检测结果** → 提示查询阶段的降级路径：
   - 无异步 → TID 通常能串联完整链路
   - 有异步 → 仍优先用 TID 查询，链路不完整时降级用 SID，最后用业务ID 模糊查询
